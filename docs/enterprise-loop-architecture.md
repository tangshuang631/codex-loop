# codex-loop 企业级 Loop 架构映射

当前结构不是最终结构。现阶段优先保证真实 loop 稳定运行，所以不会为了“目录更好看”而做无收益搬家。后续优化结构时，每次只迁移一条已经有测试保护的能力边界。

## 架构目标

`codex-loop` 的企业级演进围绕 6 个边界推进：

- Loop 内核：状态机、等待策略、预算、停止、恢复、错误分类。
- Codex 联动层：线程绑定、原生发送、聊天镜像、完成状态识别。
- NPC 决策层：Ollama 接入、规则读取、用户自定义监工角色、下一轮指令生成。
- 验证层：测试命令、构建命令、截图验收、日志检查、结果回写。
- 产品界面层：简洁中文状态、Codex 风格对话、移动端查看、用户补充引导。
- 运行治理层：本地日志、可读错误、启动关闭、长期健康检查。

## 当前文件映射

- `app/server/lib/runtime-store.mjs`
  当前仍承载最多运行状态、轮次记录、NPC 调度、验收回写和聊天镜像编排。后续应继续拆出 Loop 内核、NPC 决策、验证层和运行治理层。

- `app/server/lib/loop-controller.mjs`
  当前承载自动循环控制、开始停止和调度执行。后续应继续瘦身为 Loop 内核的编排入口。

- `app/server/lib/loop-core/controller-gates.mjs`
  自动循环闸门已经迁入 Loop 内核，负责决定是否等待 Codex、是否进入复盘、是否达到预算停止、是否允许发送下一轮。

- `app/server/lib/codex-dispatcher.mjs`
  当前承载 Codex 线程发送与可见性校验。后续应明确作为 Codex 联动层核心，且禁止不可见兜底链路伪装成功。

- `app/server/lib/codex-session-reader.mjs`
  当前承载 Codex 历史读取与聊天镜像。后续应与 dispatcher 一起收口为统一的 Codex 联动接口。

- `app/server/lib/codex-link/thread-resolver.mjs`
  目标模块。后续承载“项目路径 + Codex 窗口名”的自动匹配，只有自动匹配失败或存在歧义时才回退手动线程 ID。

- `app/server/lib/ollama-prompt-generator.mjs`
  当前承载本地模型生成、Codex 摘要和里程碑复盘。后续归入 NPC 决策层。

- `app/server/lib/ollama-loop-planner.mjs`
  当前承载创建任务阶段的规划增强。后续与 NPC 决策层共享项目记忆、文档约束和用户自定义规则。

- `app/server/lib/npc/confirmation-policy.mjs`
  已开始区分普通产品偏好与高风险人工确认，属于 NPC 决策层的一部分。

- `app/server/lib/verification/supervisor-verification.mjs`
  监督独立验收已经迁入验证层，负责安全命令过滤、验收冷却、结果摘要和失败/跳过证据注入下一轮指令。

- `app/server/lib/runtime-governance/failure-classifier.mjs`
  续跑失败分类已经迁入运行治理层，负责把 Codex 发送、本地模型、文档规则、工作区、重复发送、预算停止等异常转成中文原因与恢复动作。

- `app/server/lib/remote-access.mjs`
  当前承载远程访问能力，与移动端任务视图、设备绑定和远程引导发送同属运行治理层边界。

- `app/server/lib/runtime-governance/device-pairing.mjs`
  承载移动端扫码配对、长期绑定、令牌校验和服务重启后的自动重连基础能力。

- `app/web/src`
  桌面控制台界面层。这里应只展示用户真正需要的状态、对话和操作，不暴露工程模块概念。

- `app/mobile`
  独立移动端 App / PWA 壳。复用 Web 的任务详情数据模型，只呈现历史对话、当前状态、待合并引导和发送引导入口。

## 迁移规则

- 不得为了目录好看而搬家。只有当正在改的能力已经有测试保护，并且拆分能明显降低风险时才迁移。
- 每次只迁移一条已测试的能力边界，不一次性掏空 `runtime-store`。
- 新模块必须有明确输入输出，调用方不应依赖内部隐式状态。
- 迁移后 UI 不得显示工程模块名。模块化是为了稳定性，不是为了给用户看内部结构。
- 失败治理要先产出用户能理解的原因和下一步动作，原始错误只留在日志里。

## 生产验收入口

所有与调度、发送、验证、停止相关的迁移，都必须跑生产化检查：

```bash
npm run production:check
```

这条命令应覆盖：

- 环境检查
- Loop 本地 smoke 检查
- `npm test`
- 桌面端构建
- 移动端构建
- 前端证据检查
- git 差异检查

其中前端证据检查要确认产物中仍然包含这些关键产品信号：

- 历史对话
- 发送引导
- 截图证据
- 生产阶段
- 生产成熟度
- 剩余缺口
- 闭环证据
- 补充合并证据
- 下一轮验证
- 验证目标
- 启动预检
- 可复制的文件路径与命令
- 默认折叠的大块详情

前端证据检查本身也要写入 `runtime/frontend-evidence/`，作为生产化判断的一部分。

## 真实运行证据

代码绿灯不等于可长期使用，还需要真实运行证据。

这里的目标是形成可读的生产状态摘要，让人能直接判断是否继续长跑，而不是只看脚本是否退出成功。

- `npm run production:observe`
  读取运行日志，整理发送、等待、Codex 完成、NPC 复盘、失败和停止时间线，写入 `runtime/production-observations/`。

- `npm run production:status`
  输出生产状态摘要，汇总最近生产检查、前端证据、长跑节奏、真实运行观测、补充合并证据和下一步建议。

- `npm run production:recover`
  当 Codex 已完成但缺少 NPC 复盘时，安全补齐复盘，不启动循环，也不继续发下一轮。

真实长跑至少 2 轮闭环才算有基础证据，可以概括成：发送 -> Codex 完成 -> NPC 复盘。

默认超过 12 小时的检查或观察结果会被视为过期，需要重新运行 npm run production:check。

## 当前 P0 拆分方向

1. 先把 App 联动远程操控抬到 P0，让 `/mobile` 和 `app/mobile` 都能稳定查看循环进程、历史对话、待合并引导，并支持远程发送、编辑、撤回补充。
2. 统一桌面端与移动端的 Codex 风格对话渲染：连续对话流、浅灰分隔线、少卡片，大块文件改动、命令输出、脚本内容、测试日志和截图证据默认折叠，点击后再展开详情。
3. 强化 Loop 闸门：Codex 未完成时绝不追发，用户补充必须排队合并，达到预算后不再派发下一轮。
4. 收口 Ollama / NPC 链路：默认在本地可用时启用，真正承担产品经理、测试人员和真实用户视角的独立思考。
5. 完成线程自动绑定产品化：优先让用户通过项目路径和窗口名完成绑定，弱化手动线程 ID。

## 完成标准

一次边界迁移只有同时满足下面条件，才算完成：

- 行为测试覆盖原有能力和关键异常路径。
- 真实运行日志仍能解释失败原因。
- 桌面端和移动端仍保持纯中文、简洁、用户视角。
- 桌面端、`/mobile` 和独立移动端继续使用同一套对话数据模型。
- 大块文件、命令、脚本、测试和截图证据默认折叠，展开后可复制关键路径和命令。
- 移动端仍能查看进程、发送/编辑/撤回引导，并遵守“不打断 Codex 当前轮”的同一套闸门。
