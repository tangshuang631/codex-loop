# codex-loop Loop 工程原则

`codex-loop` 的目标不是把提示词写得更长，也不是做一个“更强模型”的包装层，而是把 AI 编程任务组织成可执行、可验证、可停止、可追踪的闭环，最终成为真正可长期使用的企业级 loop 软件。

## 核心判断

AI 编程是否好用，不只取决于模型强不强，更取决于模型外层内核、工具系统、项目记忆、验证流程、权限边界和停止条件。

单条 prompt 只能表达目标；企业级 loop 必须把目标变成持续推进机制：

```text
目标 -> 计划 -> 执行 -> 观察 -> 验证 -> 反馈 -> 下一步
```

验证通过就停止。验证失败就带着失败证据重新计划。触碰权限、成本、安全或产品边界时，必须暂停并让用户决策。

## 对 codex-loop 的产品含义

`codex-loop` 不是“自动给 Codex 发送下一句”的脚本，而是一个长期项目监工。

- 它要读懂每个项目的目标、规则、文档和验收标准。
- 它要等 Codex 完成当前轮并回到可接收指令状态后，再生成下一步。
- 它要在 Ollama 或其他本地模型可用时，承担产品经理、测试人员和真实用户视角的复盘工作，但不能越过安全边界。
- 它要把 Codex 回复、codex-loop 指令、用户补充、验证结果和异常原因沉淀为可追踪历史。
- 它要让用户在桌面端和移动端都能快速看懂“当前在做什么、为什么在等、下一步会做什么”。

产品表达也要逐步从“新建 loop”收口到“新建任务”。任务可以开始循环，也可以不开始循环；不开始循环时就进入监控模式，仍然支持查看历史和发送引导。绑定能力也要产品化，优先让用户通过项目路径和窗口名完成绑定，而不是长期依赖手动线程号。桌面端和移动端 App 都应遵守同一套任务语义与等待闸门；移动端 App 首次通过扫码完成长期绑定，之后复用受保护设备身份继续访问任务。

## 产品化 Loop 工作契约

后续所有实现都必须遵守这份契约，而不是退化成一串越来越长的续跑提示词。

- 模型只负责生成，Agent 内核负责推进。任何要发给 Codex 的下一轮内容，都必须先经过状态机、等待策略、权限边界和停止条件的过滤。
- 工具系统必须提供真实环境反馈。Codex 线程、运行日志、测试命令、构建命令、截图验收、本地模型状态都要能回到 loop 上下文。
- 项目记忆优先于临场发挥。开发清单、项目规则、用户自定义 NPC 要求、验收标准和历史决策必须进入下一轮判断。
    - 每一轮必须沉淀一组证据。至少记录 Codex 最新回复、codex-loop 发出的真实下一步、是否经过本地模型、是否处于等待、验证结果和异常原因。
    - 预算不是建议值，而是下一轮发送闸门。时间、token、失败次数、手动停止或停止条件到达后，系统可以完成当前收尾，但不能继续派发新任务。
- 用户补充不是立即打断 Codex。桌面端或移动端的补充先进入待合并队列，等 Codex 当前轮完成后再交给 NPC 合并成下一条指令。
- 监控模式不是空状态。不启动自动循环时，任务仍然是 Codex 的观察与引导入口：要能同步历史、展示状态、接收补充，并在安全时机推动一次发送。
- 没有验证信号时必须降级为人工确认或只读观察，不能为了维持循环而让 Agent 自己“猜完成了”。
- 高风险动作必须暂停。删除、覆盖、提交推送、生产资源、核心规则变更、权限边界变化，都不能由 NPC 自动拍板。

## 好 Loop 的最低标准

每个真实项目 loop 至少要具备这些部件：

- 明确目标：任务要有可理解的产品目标，而不是“你自己看着办”。
- 项目记忆：规则文档、开发清单、用户偏好、验收要求必须进入上下文。
- 工具体系：能读文件、看日志、同步 Codex 线程、调测试和构建命令、查询本地模型状态。
- 验证流程：完成不能只靠语言总结，要尽量有测试、构建、日志、截图或人工验收信号。
- 权限边界：删除、提交、推送、生产资源、高风险规则变更时必须停下来。
- 停止条件：时间、token、失败次数、用户停止、验证失败、上下文缺失，都必须能让 loop 安全停下。
- 可见历史：用户要能看到 Codex 最新回复，以及 codex-loop 准备发出的真实下一步。

## 坏 loop 比坏 prompt 更危险

坏 prompt 最多给出一次不理想回答；坏 loop 会持续消耗 token、持续改乱项目、持续把错误方向越推越远。

以下情况必须视为不合格 loop：

- 没有验收标准却持续自动续跑。
- Codex 还在工作时继续追发下一条，打断当前任务。
- 文档、项目路径或线程绑定缺失时仍然尝试推进。
- 本地模型失败后不告知用户，却假装增强成功。
- 首页只展示调试字段，不展示用户能理解的状态和历史。
- 没有停止和收尾语义，只能强行杀进程。

## NPC / 本地模型职责

本地模型不是替用户随意拍板，而是把用户目标、项目规则和 Codex 回复整理成高质量下一步。

它应该做：

- 判断 Codex 是否真的完成了当前一轮。
- 当 Codex 询问普通产品偏好时，基于文档和用户设定给出合理决策。
- 当涉及高风险操作时，要求停下来让用户确认。
- 把用户在桌面端或移动端补充的话合并进下一轮，而不是机械转发。
- 定期做产品经理、测试人员和真实用户视角的独立验收，但不要过度频繁。

它不应该做：

- 在 Codex 未完成时抢发指令。
- 忽略项目规则和开发清单。
- 为了继续循环而编造进展。
- 把冗长复述原样发给 Codex 浪费上下文。

## 企业级结构目标

为了最终成为可长期生产使用的 loop 软件，项目结构要围绕这 6 层持续收口：

1. Loop 内核：状态机、等待策略、预算、停止、恢复、错误分类。
2. Codex 联动层：线程绑定、原生发送、聊天镜像、完成状态识别。
3. NPC 决策层：Ollama 接入、规则读取、用户自定义角色、下一轮指令生成。
4. 验证层：测试命令、构建命令、截图、日志检查、结果回写。
5. 产品界面层：简洁中文状态、Codex 风格对话、移动端查看、用户补充引导。
6. 运行治理层：本地日志、可读错误、服务关闭、长期健康检查。

## 项目结构收口路线

当前代码还没有完全迁移到最终结构，但后续要持续围绕 `app/server/lib` 收口，逐步把边界拆清楚，而不是继续把所有能力都堆在同一个入口里。

建议的目标边界：

- `app/server/lib/loop-core`：状态机、等待策略、预算、停止、恢复和错误分类
- `app/server/lib/codex-link`：线程绑定、原生发送、聊天镜像、完成状态识别
- `app/server/lib/npc`：Ollama 接入、规则读取、角色设定、下一轮指令生成
- `app/server/lib/verification`：测试命令、构建命令、截图、日志检查、结果回写
- `app/server/lib/runtime-governance`：启动关闭、可读错误、长跑健康检查、日志治理

这条路线的目标不是“为了目录好看”，而是为了让 loop 的核心链路更稳定、更可验证，也更适合长期生产使用。

## 开发优先级判断

以后任何新能力都先回答三个问题：

1. 它让 loop 更可执行、更可验证，还是只是增加配置项？
2. 它减少了用户判断成本，还是把内部调试信息转嫁给用户？
3. 它失败时能不能安全停止并留下可读证据？

如果答案不清晰，就不应优先实现。
