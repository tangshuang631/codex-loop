# codex-loop Loop 工程原则

`codex-loop` 的目标不是把提示词写得更长，也不是做一个更强模型的包装层。它要成为企业级 loop 软件：把 AI 编程任务组织成可执行、可验证、可停止、可追溯的闭环。

## 核心判断

AI 编程的体验不只由模型能力决定。真正决定长期可用性的，是模型外层内核、工具系统、项目记忆、验证流程、权限边界和停止条件。

单条 prompt 只能表达目标。企业级 loop 必须把目标变成持续推进机制：

```text
目标 -> 计划 -> 执行 -> 观察 -> 验证 -> 反馈 -> 下一步
```

验证通过就停止。验证失败就带着失败证据重新计划。触碰权限、成本、安全或产品边界时，必须暂停并让用户决策。

## 对 codex-loop 的产品含义

`codex-loop` 不应该只是“自动给 Codex 发下一句话”。它需要承担一个长期项目监工的角色：

- 读懂用户为每个项目配置的目标、规则、文档和验收标准。
- 等 Codex 完成当前轮并回到可接收指令状态后，再生成下一步。
- 用 Ollama 或其它模型做 NPC / 产品经理 / 测试人员式复盘，但不能盲目覆盖安全边界。
- 把 Codex 回复、codex-loop 指令、用户临时补充、验证结果和异常原因都沉淀为可追踪历史。
- 让用户在桌面端和移动端都能看懂“当前在做什么、为什么等待、下一步会发生什么”。

产品表达必须从“新建 loop”逐步收口为“新建任务”。任务是用户真正理解的对象：它可以开启自动循环，也可以不开始循环而进入监控模式。在监控模式下，codex-loop 仍然要同步 Codex 对话、展示当前状态，并允许用户发送引导，等 Codex 当前轮完成后再交给 NPC 合并处理。

任务绑定也必须逐步产品化。优先让用户输入项目路径和 Codex 窗口名，由系统自动匹配对应窗口和线程；只有自动匹配失败或存在歧义时，才退回手动线程 ID。

## 产品化 Loop 工作契约

`codex-loop` 后续所有实现都必须遵守这个工作契约，而不是把功能退化成一条更长的续跑 prompt。

- 模型只负责生成，Agent 内核负责推进。生成结果必须经过状态机、等待策略、权限边界和停止条件过滤后才允许发送。
- 工具系统必须提供真实环境反馈。Codex 线程、运行日志、测试命令、构建命令、截图验收、本地模型状态都要能回到 loop 上下文。
- 项目记忆必须优先于临场发挥。开发清单、项目规则、用户自定义 NPC 要求、验收标准和历史决策都应该进入下一轮判断。
- 每一轮必须沉淀一组证据。至少要记录 Codex 最新回复、codex-loop 发出的下一步、是否经过本地模型、是否等待完成、验证结果和异常原因。
- 预算不是建议值，而是下一轮发送闸门。时间、token、失败次数、手动停止、停止条件到达后，系统可以完成当前收尾，但不能继续派发新任务。
- 用户补充不是立即打断 Codex。用户在控制台或移动端补充的话先进入待合并队列，等 Codex 当前轮完成后再交给 NPC 合并成下一条指令。
- 监控模式不是空状态。用户不开始循环时，任务仍应作为 Codex 的观察和引导入口，支持查看历史、发送引导、等待 Codex 可接收后合并发送。
- 没有验证信号时必须降级为人工确认或只读观察。不能因为想保持循环运行，就让 Agent 自己猜“已经完成”。
- 高风险动作必须暂停。删除、强制覆盖、提交推送、生产资源、核心规则变更、权限边界变化，都不能由 NPC 自动拍板。

## 好 loop 的最低标准

每个真实项目 loop 至少要具备这些部件：

- 明确目标：任务要有可理解的产品目标，而不是“你自己看着办”。
- 项目记忆：规则文档、开发清单、用户偏好、验收要求必须进入上下文。
- 工具系统：能读文件、看日志、同步 Codex 线程、调用验证命令、查询本地模型状态。
- 验证流程：完成不能只靠语言总结，必须尽量有测试、构建、日志、截图或人工验收信号。
- 权限边界：涉及删除、提交、推送、生产资源、高风险规则变更时必须停下来。
- 停止条件：时间、token、失败次数、用户停止、验证失败、上下文缺失都必须能让 loop 安全停止。
- 可见历史：用户要能看到最近 Codex 回复和 codex-loop 发出的真实下一步指令。

## 坏 loop 比坏 prompt 更危险

坏 prompt 最多得到一次不理想回答。坏 loop 会持续消耗 token、持续修改项目、持续扩大错误方向。

以下情况必须被视为不合格 loop：

- 没有验收标准却持续自动续跑。
- Codex 还在工作时继续追发下一条，打断当前任务。
- 文档、项目路径、线程绑定缺失时仍然尝试推进。
- 本地模型失败后不告知用户，假装智能增强成功。
- 只显示调试字段，不显示用户能理解的状态和历史。
- 没有停止和收尾语义，只能强行杀进程。

## NPC / 本地模型职责

本地模型不是替用户随意拍板，而是把用户目标、项目规则和 Codex 回复整理成下一条高质量指令。

它应该做：

- 根据 Codex 最新回复判断任务是否真正完成一轮。
- 当 Codex 请求普通产品偏好时，基于文档和用户设定给出合理决定。
- 当涉及高风险操作时，要求停下来给用户确认。
- 把用户在移动端或控制台补充的话合并进下一轮，而不是机械复制。
- 定期做产品经理、测试人员、真实用户视角的独立验收，但不要过度频繁。

它不应该做：

- 在 Codex 未完成时抢发指令。
- 忽略项目规则和开发清单。
- 为了继续循环而强行编造进展。
- 把大段无意义复述发给 Codex 浪费上下文。

## 企业级结构目标

为了最终成为可长期生产使用的 loop 软件，项目结构应围绕这些层次继续收口：

- Loop 内核：状态机、等待策略、预算、停止、恢复、错误分类。
- Codex 联动层：线程绑定、原生发送、聊天镜像、完成状态识别。
- NPC 决策层：Ollama 接入、项目规则检索、用户自定义角色、下一轮指令生成。
- 验证层：测试命令、构建命令、页面截图、日志检查、验收结果回写。
- 产品界面层：简洁中文状态、Codex 风格对话、移动端查看、用户补充引导。
- 运行治理层：本地日志、可读错误、服务关闭、长期运行健康检查。

这意味着后续开发优先级应始终按这个顺序判断：

1. 核心链路是否真实可用。
2. 是否能等待 Codex 完成后再继续。
3. 是否能基于项目文档和用户规则生成下一步。
4. 是否能验证、记录和解释结果。
5. 是否能安全停止。
6. 前端是否只展示用户真正需要的信息。

## 项目结构收口路线

当前代码仍以 `app/server/lib` 和 `app/web/src` 为主要承载目录。后续不应为了“看起来架构化”一次性大搬家，而应在修改具体功能时逐步把边界收清楚。

建议收口方向：

- `app/server/lib/loop-core`：承载状态机、等待策略、预算、停止、恢复和错误分类。
- `app/server/lib/codex-link`：承载线程绑定、原生发送、聊天镜像、完成状态识别。
- `app/server/lib/npc`：承载 Ollama 接入、项目规则读取、用户自定义监督员、下一轮指令生成。
- `app/server/lib/verification`：承载测试命令、构建命令、截图验收、日志检查和结果回写。
- `app/server/lib/runtime-governance`：承载运行日志、可读错误、启动关闭、长期健康检查。
- `app/web/src`：只保留产品界面层，负责状态展示、Codex 风格对话、移动端查看和用户补充引导。
- `app/mobile`：承载独立移动端 App/PWA 壳，复用 Web 任务详情的数据结构，只呈现历史对话、当前状态、待合并引导和发送引导入口；后续原生封装只补系统级能力。

迁移规则：

- 每次只迁移正在修改的功能边界，不做无行为收益的大重构。
- 迁移前必须有测试覆盖当前行为，迁移后测试仍然通过。
- UI 不展示内部模块名；模块化是为了稳定性，不是把工程概念暴露给用户。
- 运行失败必须先分类为用户能理解的原因和恢复动作，再暴露给首页、移动端和日志；原始错误只作为排查证据，不替代产品文案。
- `runtime/`、`settings/`、`config.local.json` 仍然只保存本机状态和用户配置，不进入仓库通用逻辑。

## 面向用户的完成定义

`codex-loop` 只有满足下面状态，才可以被称为接近生产可用：

- 用户可以创建多个项目和多个 loop，并清楚知道当前选中哪个。
- 用户可以绑定真实 Codex 线程，并在 codex-loop 里看到完整对话镜像。
- 开始循环后，系统只在 Codex 完成当前轮后发送下一条。
- 开启 Ollama 后，下一条指令会经过 NPC 工作流处理，并能使用项目文档、开发规则和用户补充。
- 用户可以随时补充下一轮引导，补充内容在发送前可编辑、可删除。
- 用户不开始循环时也可以在监控模式查看 Codex 历史并发送引导。
- 停止会进入收尾，不再发送新任务；关闭服务会提示风险并可彻底退出。
- 出错时只展示可理解错误和下一步处理建议，不把开发者调试噪音推给用户。
- 移动端能快速查看当前任务状态、最新对话和是否需要用户介入。
- 移动端 App 首次通过扫码与这台电脑长期绑定；codex-loop 进程重启后，已绑定 App 不需要重复扫码即可重连。

## 开发规则

以后任何新增功能都应先回答三个问题：

1. 它让 loop 更可执行、更可验证，还是只是增加配置项？
2. 它是否减少用户判断成本，还是把内部调试信息转嫁给用户？
3. 它失败时能不能安全停止并留下可读证据？

如果答案不明确，不应优先实现。
