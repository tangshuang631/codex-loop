# codex-loop 6.7 13:29 开发清单

## 当前阶段目标
- 把 `codex-loop` 从“开发者工具感”继续推进到“可长期使用的产品形态”。
- 优先保证后端核心链路稳定、智能、安全，再继续收口前端体验与信息架构。
- 让用户主界面只看到最关键的信息：`loop 状态`、`聊天记录`、`最近改动 / 下一步`、`创建 / 管理主内容区`。
- 产品表达优先改成“新建任务”。任务可以开启自动循环，也可以不开始循环而进入监控模式，用来查看 Codex 历史并发送引导。

## 企业级 loop 软件目标
- `codex-loop` 的最终目标不是写更长 prompt，也不是只接一个更强模型，而是成为可执行、可验证、可停止的闭环系统。
- 后续优先建设 loop，而不是堆 prompt；prompt 只是入口，真正的产品能力来自 Agent 内核、工具系统、项目记忆、验证流程、权限边界和停止条件。
- 每一轮都必须能回答：当前目标是什么、Codex 是否完成、本地模型如何判断下一步、验证证据是什么、什么时候必须停。
- 后续开发优先围绕 6 层结构推进：
- Loop 内核：状态机、等待策略、预算、停止、恢复、错误分类。
- Codex 联动层：线程绑定、原生发送、聊天镜像、完成状态识别。
- NPC 决策层：Ollama 接入、项目规则检索、用户自定义角色、下一轮指令生成。
- 验证层：测试命令、构建命令、页面截图、日志检查、验收结果回写。
- 产品界面层：简洁中文状态、Codex 风格对话、移动端查看、用户补充引导。
- 运行治理层：本地日志、可读错误、服务关闭、长期运行健康检查。
- 详细原则见：`docs/loop-engineering-principles.md`。
- 移动端目标是移动端 App，而不是缩小版网页；首次通过扫码与这台电脑长期绑定，codex-loop 重启后不用重复扫码即可复用绑定。
- 绑定目标窗口时优先通过项目路径和 Codex 窗口名自动匹配，手动线程 ID 只作为兜底。

## 企业级成熟度里程碑
- P0 可运行闭环：新建任务、绑定线程、开始循环、停止收尾、控制台可见对话。
- P1 可控闭环：不开始循环时可进入监控模式；Codex 未完成时不追发，用户补充进入待合并队列，停止条件和预算能阻止下一轮。
- P2 可验证闭环：Ollama / NPC 基于文档、规则、用户补充和 Codex 最新回复生成下一步，并把测试、构建、日志、截图等验收结果写回状态。
- P3 可长期监控闭环：日志可读、失败可恢复、移动端 App 可查看进展并补充下一轮引导，支持扫码长期绑定。
- P4 企业级治理闭环：多项目多 loop 稳定运行，角色规则可定制，权限边界、成本、失败率和质量趋势可审计。

## 后端核心链路重点

### 1. loop 智能续发
- 目标：下一条自动续发消息不只是模板拼接，而是基于当前上下文动态生成。
- 重点输入：
- 最近摘要
- 当前任务阶段
- 风险状态
- 文档规则
- 已有 Codex 回复
- 目标效果：
- 更像真人继续推进
- 更少重复和机械续发
- 在异常状态下能改变语气与策略，而不是盲推

### 2. NPC / 规划器增强
- 目标：把“新建任务”从简单问答升级为更完整的任务规划流程。
- 目标流程：
- 识别项目
- 检查 git / docs / 可用命令
- 生成计划
- 风险确认
- 创建任务
- 通过项目路径和窗口名自动匹配 Codex 线程
- 需要继续加强：
- 规划结果可追溯
- 风险项更明确
- 对缺少 git / 缺少规则文档时的处理更稳

### 3. 任务监控模式
- 目标：任务不等于自动循环，不开始循环也能作为 Codex 观察和引导入口。
- 目标行为：
- 同步绑定窗口的历史对话
- 显示当前 Codex 是处理中、等待下一轮还是需要人工处理
- 支持发送引导，但不打断 Codex 当前轮
- 引导先进入待发送队列，等 Codex 完成后交给 NPC 合并
- 自动循环关闭时不主动派发下一轮，只按用户显式发送引导推进
- 当前已接入：停止后的绑定任务会显示为监控中，桌面端可把底部补充保存为待发送引导，再从补充气泡的小按钮编辑、撤回或手动发送一次；移动端已绑定设备发送引导时也会复用同一条监控模式手动派发链路。Codex 正在处理或本地模型复盘时只排队等待，不会打断当前轮；线程发送失败会直接提示错误，不伪装成已排队。手机端已支持在合并前编辑或撤回待合并引导，避免远程误输入后无法收回。

### 4. 安全护栏
- 目标：保证自动化 loop 在异常场景下“安全地停”而不是“盲目继续”。
- 必须覆盖的情况：
- 权限不足
- git 缺失
- 文档缺失
- 本地模型异常
- 自动续发失败
- 目标行为：
- 暂停 loop
- 明确告诉用户原因
- 给出下一步建议
- 保留可追溯记录

## 前端产品化重点

### 1. 主界面收口
- 主界面只保留：
- 当前任务状态
- 聊天记录
- 最近改动 / 下一步
- 创建 / 管理切换后的主内容区
- 不再在主界面堆大量引导性说明和开发测试味文案。

### 2. Help 页
- 所有解释性文字、引导、术语说明统一迁移到 `Help` 页。
- 主界面只保留“操作必需信息”。
- Help 页应包含：
- loop 是什么
- 如何创建任务
- 如何绑定线程
- 如何开启本地模型增强
- 如何处理 git / 权限 / 文档问题
- 如何安全停止与关闭服务

### 3. 信息层级与折叠
- 普通配置尽量折叠。
- 只把关键操作露出来。
- 目标风格：
- 高信息密度
- 不拥挤
- 更接近 Codex 的桌面产品风格
- 重点关注：
- 侧边栏只做导航
- 右侧主内容区按功能整块切换
- 创建任务不再挤在侧边栏

### 4. 移动端 App
- 移动端不是完整后台页，而是远程监控和轻量引导工具。
- 核心界面：
- 当前任务状态
- 历史对话
- 发送引导输入框
- 待发送引导队列
- 首次使用通过扫码与当前电脑长期绑定；服务重启后复用绑定，不要求重复扫码。
- 高风险操作仍必须回到桌面端或明确人工确认，移动端不做破坏性快捷操作。
- 当前状态：桌面端已具备扫码配对和长期令牌校验基础，`/mobile` 轻量入口已支持查看任务、历史对话、发送引导、编辑和撤回待合并引导，并在安全时远程手动推进一次；独立 `app/mobile` 移动端 App/PWA 壳已建立，继续聚焦远程监控、历史对话和轻量引导，后续再补原生封装。

- 已完成并验证：
- `npm test`
- `npm run build:web`
- `npm run build:mobile`
- 生产就绪检查入口：`npm run production:check`，会顺序执行环境检查、长跑 smoke 检查、测试、桌面端构建、移动端构建、前端证据检查和 git 差异检查，并写入 `runtime/production-checks/` 报告。前端证据检查会确认构建产物仍包含历史对话、发送引导和截图证据，并把结果写入 `runtime/frontend-evidence/`。
- 生产状态摘要入口：`npm run production:status`，会读取最近生产检查、前端证据、长跑节奏和真实运行观测报告，输出当前是否可继续使用以及下一步建议；它只读本地报告，不启动循环也不向 Codex 发送消息。摘要会额外输出 `readiness.stage`：`trial` 表示代码闸门通过但缺少真实 2 轮闭环证据，`observing` 表示真实任务正在等待 Codex 或 NPC 复盘，`production` 表示代码闸门和真实 2 轮闭环证据都通过，`blocked` 表示必须先处理失败项。默认超过 12 小时的报告会被视为已过期，需要重新运行 npm run production:check 后再判断是否适合长期运行。
- 真实运行观测入口：`npm run production:observe`，会读取真实任务的本地运行日志，整理发送、等待、Codex 完成、NPC 复盘、失败和停止时间线，并写入 `runtime/production-observations/`；它只读日志，不启动循环也不向 Codex 发送消息，用于补足本地 smoke 无法证明的长期运行证据。
- 长期运行基本证据要求至少 2 轮连续真实闭环：发送下一轮指令 -> Codex 完成 -> NPC 复盘。只有 1 轮闭环时只说明链路可试用，还不能证明适合提高自动化时长。
- 生产恢复入口：`npm run production:recover`，用于 Codex 已有完成回复但缺少 NPC 监督复盘的情况；它只补齐监督复盘，不启动循环，也不会发送下一轮指令。
- 长跑 smoke 检查入口：`npm run loop:smoke`，只做本地模拟，不发送真实 Codex 消息；它验证发送后等待 Codex、未完成不追发、用户补充会等 Codex 完成后交给 NPC 合并、完成后先监督复盘、按产品经理 / 测试人员 / 真实用户视角做独立验收并遵守冷却防重复、预算到达后停止，是长期运行前的节奏护栏。

## 接下来主要开发方法

### 方法 1：核心链路优先
- 先修和增强后端主链路，不先做大重构。
- 目标是尽快让产品“更聪明、更稳、更安全”。

### 方法 2：TDD + 回归验证
- 先补失败测试
- 再补实现
- 最后统一跑：
- `npm run production:check`
- `npm run loop:smoke`
- `npm test`
- `npm run build:web`
- `npm run build:mobile`

### 方法 3：真实用户视角验收
- 不只看代码和接口。
- 要从用户视角看：
- 页面是否清晰
- 操作是否顺手
- 信息是否拥挤
- 是否还有开发者味内容

### 方法 4：主界面减噪
- 一切非核心说明文案优先迁移到 `Help` 页。
- 主界面只承载高频核心任务。

## 本轮最重要的产品判断标准
- 智能：续发更像真人，不机械。
- 快捷：创建、切换、查看状态足够直接。
- 安全：异常时会停、会提示、可追溯。
- 清爽：主界面没有多余解释和开发者输出。
- 稳定：可长时间运行，不容易让用户误判为卡死。
