# codex-loop 6.7 13:29 开发清单

- 把 `codex-loop` 从“开发者工具感”继续推进到“可长期使用的产品形态”。
- 优先保证后端核心链路稳定、智能、安全，再继续收口前端体验与信息架构。
- 让用户主界面只看到最关键的信息：`loop 状态`、`聊天记录`、`最近改动 / 下一步`、`创建 / 管理主内容区`。

1. 先补后端 `loop / NPC / 大模型` 核心链路。

3. 最后继续做前端产品化与真实交互验收，向 Codex 风格靠拢。

## 后端核心链路重点

### 1. loop 智能续发
- 目标：下一条自动续发消息不只是模板拼接，而是基于当前上下文动态生成。
- 重点输入：
- 最近摘要
- 当前任务阶段
- 风险状态
- 文档规则
- 已有 Codex 回复
- 目标效果：
- 更像真人继续推进
- 更少重复和机械续发
- 在异常状态下能改变语气与策略，而不是盲推

### 2. NPC / 规划器增强
- 目标：把“创建 loop”从简单问答升级为更完整的任务规划流程。
- 目标流程：
- 识别项目
- 检查 git / docs / 可用命令
- 生成计划
- 风险确认
- 创建 loop
- 需要继续加强：
- 规划结果可追溯
- 风险项更明确
- 对缺少 git / 缺少规则文档时的处理更稳

### 3. 安全护栏
- 目标：保证自动化 loop 在异常场景下“安全地停”而不是“盲目继续”。
- 必须覆盖的情况：
- 权限不足
- git 缺失
- 文档缺失
- 本地模型异常
- 自动续发失败
- 目标行为：
- 暂停 loop
- 明确告诉用户原因
- 给出下一步建议
- 保留可追溯记录

## 前端产品化重点

### 1. 主界面收口
- 主界面只保留：
- 当前 loop 状态
- 聊天记录
- 最近改动 / 下一步
- 创建 / 管理切换后的主内容区
- 不再在主界面堆大量引导性说明和开发测试味文案。

### 2. Help 页
- 所有解释性文字、引导、术语说明统一迁移到 `Help` 页。
- 主界面只保留“操作必需信息”。
- Help 页应包含：
- loop 是什么
- 如何创建任务
- 如何绑定线程
- 如何开启本地模型增强
- 如何处理 git / 权限 / 文档问题
- 如何安全停止与关闭服务

### 3. 信息层级与折叠
- 普通配置尽量折叠。
- 只把关键操作露出来。
- 目标风格：
- 高信息密度
- 不拥挤
- 更接近 Codex 的桌面产品风格
- 重点关注：
- 侧边栏只做导航
- 右侧主内容区按功能整块切换
- 创建任务不再挤在侧边栏

## 接下来主要开发方法

### 方法 1：核心链路优先
- 先修和增强后端主链路，不先做大重构。
- 目标是尽快让产品“更聪明、更稳、更安全”。

### 方法 2：TDD + 回归验证
- 先补失败测试
- 再补实现
- 最后统一跑：
- `npm test`
- `npm run build:web`

### 方法 3：真实用户视角验收
- 不只看代码和接口。
- 要从用户视角看：
- 页面是否清晰
- 操作是否顺手
- 信息是否拥挤
- 是否还有开发者味内容

### 方法 4：主界面减噪
- 一切非核心说明文案优先迁移到 `Help` 页。
- 主界面只承载高频核心任务。

## 本轮最重要的产品判断标准
- 智能：续发更像真人，不机械。
- 快捷：创建、切换、查看状态足够直接。
- 安全：异常时会停、会提示、可追溯。
- 清爽：主界面没有多余解释和开发者输出。
- 稳定：可长时间运行，不容易让用户误判为卡死。
