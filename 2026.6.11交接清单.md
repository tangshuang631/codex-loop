## 当前核心链路进度

### 已完成
- 任务模型已经从“loop”逐步产品化为“任务”。
- 任务支持两种模式：
- 自动循环：等待 Codex 完成后，再由 NPC / Ollama 生成下一轮指令。
- 监控模式：不自动循环，也能同步 Codex 历史、查看状态、发送一次性引导。
- Codex 未完成时不会追发下一条，避免打断正在执行的 Codex。
- 用户补充会先进入待合并队列，等 Codex 完成后再交给 NPC / Ollama 合并。
- 停止条件、预算、收尾、运行状态、失败恢复、生产检查入口已接入。
- 启动器已经处理过旧进程清理、端口、工作区默认值和启动状态残留问题。
- 生产状态系统已经能区分：
- 代码闸门是否通过
- 是否有真实闭环证据
- 是否有用户补充合并证据
- 是否可以短时试用
- 是否可以长跑

## 接手建议顺序
1. 先运行 `git status -sb`，确认工作区干净。
2. 运行 `npm run production:status`，看成熟度和缺口。
3. 如果代码闸门过期，运行 `npm run production:check`。
4. 启动 codex-loop 服务，打开桌面端。
5. 用当前已绑定任务跑 `1` 轮真实循环。
6. 插入一条用户补充，等 NPC / Ollama 合并进下一条。
7. 再运行 `npm run production:status`，看是否达到长期运行基础证据。
8. 真机访问 `/mobile-app` 做移动端验收。

## 当前不要误判的点
- `75%` 不等于没法用；它表示代码闸门通过，适合短时真实试用。
- `75%` 也不等于成熟生产可长跑；缺的是真实运行证据，不是单纯再写几个单测。
- Codex 桌面端不一定实时展示 codex-loop 的派发内容；当前产品策略是优先在 codex-loop 的对话流里完整观察。
- 用户补充不应该立即打断 Codex；正确行为是先显示“待下一轮合并”。
- 移动端目前是 PWA / App 壳能力，后续原生 App 封装仍要继续。
