# codex-loop

`codex-loop` 是一个面向真实项目开发的本地任务控制台。  
它不是替代 Codex 桌面端，而是把“持续推进、可见状态、历史对话、优雅停止、移动端查看、用户补充引导”这些长期使用最关键的能力补齐，让你能更稳地把 Codex 用在项目开发里。

核心原则见：

- [Loop 工程原则](docs/loop-engineering-principles.md)
- [企业级 Loop 架构映射](docs/enterprise-loop-architecture.md)

`codex-loop` 的长期目标不是做一个“更强模型”的包裹层，而是做成一个企业级 loop 软件：把 AI 编程任务变成可执行、可验证、可停止、可追踪的闭环。

## 它解决什么问题

- 循环开始后，不确定到底有没有真的发出去。
- 长任务跑着跑着就看不懂当前在做什么。
- Codex 回复、codex-loop 指令、用户补充、验证结果分散在不同地方。
- 想让任务持续推进，但又不希望在 Codex 还没做完时被打断。
- 想在手机上远程看进度、看历史、补一句下一轮引导。

## 当前产品方向

- 首页只保留最关键的信息：这一轮在做什么、下一步该做什么、最常用入口是什么。
- 任务详情以连续对话流为主，尽量接近 Codex 桌面端。
- 大块文件改动、命令输出、脚本内容、测试日志、截图证据默认折叠，需要时再展开。
- 用户补充不会立即打断 Codex，而是进入待合并队列，等当前轮完成后再交给本地监督流程合并。
- 移动端查看与发送引导是 `P0` 能力，不是附属页面。

## 你可以把它当成什么

- 一个面向真实项目开发的本地循环控制台
- 一个让长任务更可见、更可控的产品层
- 一个把 Codex 对话、状态、引导和验证整合在一起的辅助系统

## 当前能力边界

- 不接入官方内部私有接口
- 不承诺完全无人值守开发
- 不绕过权限、提交、推送、生产资源这类真实约束
- 本地模型增强当前优先支持 Ollama

## 企业级结构

后续架构会持续收口到这 6 层：

1. Loop 内核：状态机、等待策略、预算、停止、恢复、错误分类
2. Codex 联动层：线程绑定、原生发送、聊天镜像、完成状态识别
3. 监督决策层：本地模型接入、规则读取、角色设定、下一轮指令生成
4. 验证层：测试命令、构建命令、截图证据、日志检查、结果回写
5. 产品界面层：简洁中文状态、Codex 风格历史对话、移动端查看、发送引导
6. 运行治理层：本地日志、可读错误、启动关闭、长期健康检查

## 目录结构

```text
app/
  server/        本地接口、循环控制、线程联动、模型与运行治理
  web/           桌面控制台
  mobile/        独立移动端 App / PWA 壳
  shared/        桌面端与移动端共享展示逻辑
docs/            产品原则、架构文档、设计说明
projects/        通用项目适配模板
scripts/         启动、检查、验证、观察、恢复脚本
tests/           核心测试
runtime/         本地运行产物（git 忽略）
settings/        本地状态与设备绑定信息（git 忽略）
dist/            构建产物（git 忽略）
```

## 环境要求

最低要求：

- Windows
- Node.js 18 或更高版本
- npm
- 可正常使用的 Codex 桌面端 / CLI 工作环境

建议准备：

- Git
- 一个真实项目目录
- 一个用于持续推进的 Codex 对话窗口

可选增强：

- [Ollama](https://ollama.com/)

## 安装

1. 克隆仓库
2. 进入仓库目录
3. 执行 `npm install`

## 本地配置

仓库默认配置是通用模板，不绑定任何人的私有项目。  
你的本机配置请写到：

- `config.local.json`

建议做法：

1. 复制 `config.local.example.json` 为 `config.local.json`
2. 把 `workspaceRoot` 改成你自己的项目目录

这些配置不会进入仓库。

## 快速开始

1. `npm install`
2. `npm run production:check`
3. 双击 `start-codex-loop.bat`
4. 打开控制台页面
5. 创建项目
6. 在项目下新建任务
7. 绑定对应 Codex 窗口
8. 开始循环，或先以监控模式查看历史与发送引导

## 生产化检查

投入真实使用前，先运行：

```bash
npm run production:check
```

这条命令会顺序执行：

- 环境检查
- Loop 本地 smoke 检查
- 全量测试
- 桌面端构建
- 移动端构建
- 前端证据检查
- git 差异检查

报告会写入：

- `runtime/production-checks/`

其中前端证据检查会把结果写入 `runtime/frontend-evidence/`，确认产物里仍然保留这些关键产品信号：

- 历史对话
- 发送引导
- 截图证据
- 启动预检
- 文件改动
- 命令输出
- 默认折叠的详情块
- 可展开查看的完整详情

如果失败，先按报告中的失败项处理，再开始长期循环。

## 生产状态与观察

查看当前是否适合继续长期使用：

```bash
npm run production:status
```

这条命令输出的是生产状态摘要，会汇总最近生产检查、前端证据、长跑节奏和下一步建议。

查看真实运行观察报告：

```bash
npm run production:observe
```

真实长跑要至少形成 2 轮闭环证据，才能说明已经具备长期使用基础：

- 发送下一轮
- Codex 完成当前轮
- 监督复盘并生成下一步

当 Codex 已完成，但本地监督流程的监督复盘缺失时，可以安全补齐：

```bash
npm run production:recover
```

它只补当前轮复盘，不会继续发下一轮。

默认超过 12 小时的生产检查或观察报告会被视为过期，这时应重新运行 `npm run production:check` 再判断是否适合继续长期运行。

## 任务模式

界面统一面向“任务”，而不是要求用户先理解 loop 概念。

一个任务可以处于两种主要模式：

- 监控模式：不自动循环，只同步 Codex 历史、展示状态，并允许发送引导
- 循环模式：等待 Codex 完成当前轮后，再生成并发送下一轮

这意味着即使你不开始循环，也仍然可以通过 `codex-loop` 观察项目、看历史、补充下一步要求。新建任务后，如果不开始循环，就降级为监控模式。

## 任务创建与绑定

当前目标流程是：

1. 输入项目路径
2. 输入 Codex 窗口名
3. 系统优先自动匹配可用窗口 / 线程
4. 自动匹配失败时，再回退到手动线程 ID

这样会逐步替代“先去 Codex 里找线程号再复制回来”的方式。

## 循环运行规则

当前循环的核心约束是：

- Codex 未完成时，不继续追发
- 用户补充先排队，不直接打断
- 本地监督流程负责结合文档、规则、历史和补充整理下一轮
- 达到停止条件后，只做收尾，不再派发新任务
- 出错时优先展示用户能理解的中文原因和下一步建议

## 默认模式

即使不开启任何本地模型，`codex-loop` 也能正常工作：

- 默认中文续跑
- 根据当前任务状态继续推进
- 可查看状态、历史、摘要
- 可停止、重启、切换任务

也就是说，不装 Ollama 也可以直接开始使用。

## Ollama 增强模式

如果你希望下一轮指令更像一个真的产品经理 / 测试人员 / 监工整理出来的内容，可以安装 Ollama。

基础步骤：

1. 安装 [Ollama](https://ollama.com/)
2. 启动 Ollama
3. 拉取模型，例如：

```bash
ollama pull qwen2.5:7b
```

4. 打开控制台
5. 在设置里开启 Ollama，并选择模型

开启后，系统会优先基于这些信息整理下一轮：

- 当前任务目标
- 项目文档与开发规则
- Codex 最新回复
- 用户补充引导
- 最近一次验证或失败证据

如果本机检测不到 Ollama，系统应降级但仍可继续使用。

## 移动端与远程查看

当前移动端正在按 `P0` 推进，方向是：

- 手机上能看当前任务状态
- 手机上能看完整历史对话
- 手机上能发送、编辑、撤回待合并引导
- Codex 正在处理时只排队，不打断
- 扫码长期绑定基础已接入
- 受保护移动端任务视图会通过长期设备令牌访问
- 首次通过扫码绑定设备
- 后续服务重启后已绑定设备可自动重连
- 自动窗口绑定基础已接入，可优先通过项目路径和窗口名完成绑定

当前桌面端 `/mobile` 页面和 `app/mobile` 独立壳会共用同一套后端数据与对话模型。

当前的独立移动端 App / PWA 壳已经接入；也可以写成独立移动端 App/PWA 壳。后续真正的原生 App 主要补系统级能力，而不是重做一套历史对话界面，这部分属于原生 App 后续能力。

这里的目标不是只“远程查看”，而是支持远程发送引导、远程操控任务节奏，并保持 Codex 风格历史对话可读。

为了判断是否真的适合长时间运行，生产状态摘要至少要能回答这些问题：

- 最近生产检查是否通过
- 前端证据是否完整
- 长跑节奏是否健康
- 下一步建议是什么
- 是否需要重新运行 npm run production:check

真实闭环证据至少 2 轮才算基础有效，可以概括为：发送 -> Codex 完成 -> 监督复盘。

移动端和桌面端都会展示同一套闭环证据：当前完成了几轮真实闭环、是否已有用户补充合并证据，以及下一轮验证应该重点确认什么。用户补充会先进入待合并队列，等 Codex 完成后由本地监督流程合并进下一条指令，再形成可追踪的“已合并补充”记录。

历史对话会尽量接近 Codex 桌面端：普通回复按连续对话流展示，代码块、文件路径、命令输出、脚本内容、截图证据等细节默认折叠；需要检查时再展开，文件路径和命令支持复制，方便手机端和桌面端都能快速验收。
